# Implementation Plan

このファイルは、初回実装に入るための計画である。

実装時は `README.md` の作業別参照ルールに従い、横断的な基準は `shared-rules.md` を優先する。

## リポジトリ現状

現時点のリポジトリには、アプリケーション実装や `package.json` は存在しない。

存在する主なファイルは以下である。

* `AGENTS.md`
* `.codex/product-brief.md`
* `specs/`

そのため、初回実装では新規フロントエンドアプリケーションを作成する。

## 技術スタック

初回実装では、以下を採用する方針とする。

* Vite
* React
* TypeScript
* CSS Modules または通常の CSS

理由:

* タイムライン編集UIをコンポーネントとして分割しやすい。
* ドメインロジックを TypeScript の型で表現しやすい。
* MVP段階では、バックエンドなしで画面上の検証を進めやすい。

依存パッケージ追加が必要になるため、実装開始時にユーザー確認を行う。

## 初回実装の範囲

初回実装では、CSV取り込みや永続保存よりも、タイムライン編集体験と検証ロジックの核を優先する。

実装するもの:

* サンプルデータによる単一イベント表示
* 複数ブースのタイムライン表示
* ミニイベントと個人DJの応募一覧表示
* ミニイベント参加DJとDJ別ブース使用時間の表示
* 手動配置操作
* 同一DJ重複の強いエラー判定と表示
* 希望時間外配置の弱いエラー判定と表示
* DJ別のブース使用回数とブース使用時間の表示
* ブース利用可能時間と休憩時間の表示

## 初回実装で作らないもの

以下は初回実装では作らない。

* CSV取り込み
* Googleスプレッドシート連携
* データ保存
* データ出力
* 画像出力
* 自動配置
* 複数ユーザー編集
* 認証
* PDF出力

## 仮置きする未確定事項

初回実装ではサンプルデータを使うため、以下の未確定事項は実装をブロックしない。

* CSV列定義
* CSV文字コード
* CSV時刻表現
* 人物IDの具体的な形式
* CSV取り込み時の不整合処理
* 保存形式
* 出力形式

初回実装では、人物IDは任意の文字列として扱う。

希望時間区分の初期値は、サンプルイベント内の設定値として持たせる。

エラー表示の見た目は、以下の仮表現とする。

* 強いエラー: 赤系の表示
* 弱いエラー: 黄系の表示

## 実装順序

1. Vite + React + TypeScript の最小構成を作成する。
2. `shared-rules.md` と `domain-model.md` に基づいて TypeScript 型を定義する。
3. サンプルデータを作成する。
4. 時間計算ユーティリティを実装する。
5. 同一DJ重複と希望時間外配置の検証ロジックを実装する。
6. DJ使用状況集計ロジックを実装する。
7. タイムライン、応募一覧、エラー一覧、DJ使用状況のUIを実装する。
8. 手動配置操作を実装する。
9. 画面表示と検証結果を確認する。

## テスト方針

初回実装では、ドメインロジックを優先してテストする。

テスト対象:

* 時間範囲の重複判定
* 同一DJ重複の強いエラー判定
* 希望時間外配置の弱いエラー判定
* ミニイベント参加DJを含むDJ使用状況集計
* `usageMinutes` が集計に使われること

UIについては、初回は手動確認を中心とする。

## 実装前の確認事項

実装開始前に、以下をユーザーへ確認する。

* 新規フロントエンドとして Vite + React + TypeScript を追加してよいか。
* 初回実装は CSV 取り込みを含めず、サンプルデータで進めてよいか。
* 初回実装では永続保存・出力を含めない方針でよいか。

## 成功条件

初回実装は、以下を満たせば完了とする。

* ブース別タイムラインが画面上で確認できる。
* ミニイベントと個人DJをタイムラインに配置できる。
* 同一DJが別ブースで時間重複した場合、強いエラーとして確認できる。
* 希望時間外配置が弱いエラーとして確認できる。
* 特定DJのブース使用回数とブース使用時間を確認できる。
* ミニイベント参加DJの使用時間は、ミニイベント全体の配置時間ではなく参加DJごとの使用時間で集計される。
