## SDD Workflow

このプロジェクトでは Specification Driven Development を採用する。

### ファイルの役割

* `.codex/product-brief.md`

  * 対話的に整理したプロダクト理解を保存する作業メモ。
  * 正式仕様ではない。
  * 要件整理、仕様作成、実装計画、実装改善提案の参照元として扱う。

* `specs/`

  * 正式仕様を保存するディレクトリ。
  * ユーザーが明示的に「仕様を書き出してください」と依頼するまで作成・更新しない。

### 要件整理

* 最初から実装しない。
* まず `.codex/product-brief.md` を参照し、対話的に要件を整理する。
* 新しい合意が発生した場合は、`.codex/product-brief.md` の更新案を提示する。
* ユーザーが承認した場合、またはユーザーが明示的に更新を依頼した場合のみ、`.codex/product-brief.md` を更新する。
* 推測は確定事項として扱わず、「仮定」または「未確定事項」に分ける。

### 仕様生成

* ユーザーが「仕様を書き出してください」と明示した場合のみ、`specs/` 以下に正式仕様を生成する。
* 正式仕様は、`.codex/product-brief.md` とユーザーとの合意内容を根拠として作成する。
* 仕様生成と同時に実装しない。

### 実装

* ユーザーが「実装してください」と明示するまで実装しない。
* 実装前に、`.codex/product-brief.md` を確認する。
* `specs/` が存在する場合は、まず `specs/README.md` を確認し、作業内容に関連する正式仕様のみを参照する。
* `specs/shared-rules.md` が存在する場合、人物ID、時間単位、エラー強度、DJ使用状況集計などの横断的な基準は `specs/shared-rules.md` を優先する。
* `specs/` が存在する場合、実装時の正式な参照元は `specs/README.md` と関連仕様ファイルとする。`.codex/product-brief.md` は背景資料として扱い、正式仕様と矛盾する場合は仕様更新を提案する。
* 実装前に、変更予定ファイル、実装方針、テスト方針を提示する。
* 依存パッケージ追加、大規模リファクタリング、破壊的変更は事前に確認する。

### 実装改善

* ユーザーが「実装改善を提案してください」と明示した場合のみ、`.codex/product-brief.md` と現実装の差分を確認し、改善案を提示する。
* `specs/` が存在する場合は、まず `specs/README.md` を確認し、作業内容に関連する正式仕様のみを参照する。
* `specs/shared-rules.md` が存在する場合、横断的な基準は `specs/shared-rules.md` を優先する。
* 改善案を提示しても、ユーザーが承認するまで実装しない。
